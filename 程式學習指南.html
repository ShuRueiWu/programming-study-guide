1程式如何被執行

寫程式的本質：把人能讀的指令，變成電腦能執行的動作。中間隔著一層「翻譯」。理解這層翻譯，很多怪現象（為什麼錯誤訊息長那樣、為什麼改了沒效果）就有解釋。

編譯 (Compile) vs 直譯 (Interpret)

方式怎麼做代表語言類比
編譯整份原始碼先一次翻成機器碼（執行檔），再執行C, C++, Rust, Go把整本外文書先全部翻譯印出再給人看
直譯一行一行邊翻譯邊執行Python, JavaScript, Ruby請口譯員現場一句句翻

Python 是直譯語言（嚴格說：先轉成 bytecode 再由虛擬機執行，但你可當成「一行行跑」）。好處是改完馬上能跑、互動性高；代價是速度比編譯語言慢。

💡 關鍵心智模型

電腦極度笨但極度快。它不會「猜你的意思」，只會一字不差照你寫的做。程式出錯 99% 不是電腦壞了，是你的指令和你以為的意思有落差。Debug 的本質就是找出這個落差。

原始碼 → 執行的旅程

你寫的 .py 檔 → Python 直譯器讀取 → 逐行執行 → 輸出結果 / 報錯。當你在終端機打 python3 script.py，就是叫直譯器去跑那個檔。

🩺 醫療類比

編譯像事先排好的給藥流程表（一次定稿、跑很快、改動要重編）；直譯像查房時口頭逐項醫囑（即時、彈性、但每次都要重念一遍）。Python 屬後者——適合探索資料、快速試做。

3控制流程 (Control Flow)

程式預設由上往下一行行跑。控制流程讓它能「分岔」與「重複」——這是程式從「計算機」變成「能做決策的工具」的關鍵。

條件判斷 if / elif / else

pythontemp = 38.5
if temp >= 38.0:
    print("發燒")
elif temp < 36.0:
    print("低體溫")
else:
    print("體溫正常")
💡 Python 用「縮排」定義區塊

多數語言用 { } 框住區塊，Python 用縮排（通常 4 個空格）。同一層的程式碼縮排必須一致，否則直接報 IndentationError。這逼你寫出視覺整齊的程式——是特色不是 bug。

迴圈 (Loops)：重複做事

for：對一群東西逐一處理

pythonlabs = [120, 95, 180]
for v in labs:
    if v > 140:
        print(v, "偏高")

while：條件成立就一直做

pythonn = 3
while n > 0:
    print("倒數", n)
    n = n - 1   # 一定要讓它趨近結束
⚠️ 無窮迴圈 (Infinite Loop)

while 若忘了更新條件變數，會永遠跑不停、程式卡死。終端機按 Ctrl + C 強制中斷。for 較安全（跑完清單就停），不確定次數時才用 while。

布林邏輯與真假值

條件由比較運算子（> < == != >= <=）與邏輯運算子（and / or / not）組成。Python 還有「Truthy/Falsy」：空字串、空清單、0、None 都當成 False。所以 if labs: 等於「如果清單非空」。

4函式 (Functions)

函式是把一段會重複用的邏輯包起來、給它名字。輸入參數 → 處理 → 回傳結果。它是寫出可維護程式的最重要單一觀念。

pythondef bmi(weight_kg, height_m):
    """計算 BMI，回傳數值。"""   # docstring：說明用途
    return weight_kg / (height_m ** 2)

result = bmi(70, 1.75)    # 呼叫，result = 22.86

為什麼一定要學會用函式

  • DRY（Don't Repeat Yourself）：同樣邏輯只寫一次。改公式只改一處，不會漏改。
  • 抽象 (Abstraction)：用的時候只需知道「給什麼、回什麼」，不必每次重看內部。像呼叫檢驗——你看報告，不必親自跑質譜儀。
  • 可測試：小而單一職責的函式，容易單獨驗證對錯。

參數、回傳、作用域

概念說明
位置參數 / 關鍵字參數bmi(70, 1.75) 靠順序；bmi(height_m=1.75, weight_kg=70) 靠名字（較清楚不易錯）
預設值def f(x, unit="mg"): 沒給就用預設
return把結果丟回呼叫端。沒寫 return 則回傳 None
作用域 (scope)函式內建立的變數只活在函式內，外面看不到——這是好事，避免互相干擾
✓ 黃金準則

一個函式只做一件事，名字就是它做的事（動詞開頭：calculate_dose、load_patients）。如果一個函式要寫一段註解才解釋得清楚它在幹嘛，通常代表該拆成兩個。

5資料結構選型

用對資料結構，程式碼短一半、跑快十倍。四個最常用的，重點是「何時用哪個」。

結構寫法特性什麼時候用
list 清單[3,1,2]有序、可改、可重複一串東西、會增刪、在意順序（檢驗時間序列）
tuple 元組(25.04, 121.5)有序、不可改固定不變的一組（經緯度、RGB、回傳多值）
dict 字典{"hr":80,"bp":120}鍵→值對照、查找極快「用名字查資料」（一筆病歷、設定檔、計數）
set 集合{1,2,3}無序、不重複去重、判斷「有沒有」、交集/聯集
💡 dict 與 set 為何「快」

它們用 hash（雜湊）實作：查一筆資料幾乎是「直接算出位置」，不必從頭找。在 100 萬筆裡查一個 key，dict 幾乎瞬間；list 用 in 找則要一筆筆比對。「我要反覆查某個東西在不在」就用 set/dict，別用 list。

選擇決策樹

  • 要順序且會修改 → list
  • 要用名字/key 查 → dict
  • 只在意有沒有、要不要去重 → set
  • 一組值固定不該被改 → tuple
🩺 醫療類比

一筆病人資料用 dict（{"name":..., "age":..., "dx":...}）；一批病人用 list of dict；想知道「這些病人涵蓋哪些不重複的診斷碼」用 set。這三者組合幾乎能表達所有臨床表格資料。

6物件導向入門 (OOP)

當資料和「對這些資料的操作」總是綁在一起，就值得包成一個類別 (class)。Class 是「藍圖」，依藍圖造出來的具體東西叫物件/實例 (instance)。

pythonclass Patient:
    def __init__(self, name, weight, height):  # 建構子
        self.name = name        # 屬性 (attribute)
        self.weight = weight
        self.height = height

    def bmi(self):                # 方法 (method)
        return self.weight / (self.height ** 2)

p = Patient("Wu", 70, 1.75)   # 造一個實例
print(p.name, p.bmi())        # Wu 22.86

四個核心詞

詞意思類比
class 類別藍圖／模板「病人」這個概念
instance 實例依藍圖造出的具體物件王先生這個具體的人
attribute 屬性物件身上的資料姓名、體重
method 方法物件能做的動作（綁在它身上的函式）算 BMI
💡 self 是什麼

self 代表「這個物件自己」。每個 method 第一個參數都是 self，讓方法能存取這一個物件的資料。p.bmi() 時 Python 自動把 p 當成 self 傳進去。

✓ 初學者層級的建議

先把 class 當成「把相關資料 + 操作打包」的工具就好。繼承 (inheritance)、多型 (polymorphism) 等進階特性，等你真的遇到「兩種東西很像但有點不同」再學，不必一開始硬塞。

pythontry:

9解題流程：從問題到程式

初學者最大的卡點不是語法，是「不知道怎麼開始」。解法是：別急著打字，先拆解。

  1. 講清楚輸入與輸出：給什麼資料？要得到什麼結果？（例：給一串檢驗值 → 回傳異常的那些）
  2. 拆成小步驟（分解 Decomposition）：大問題切成能各自解決的小問題。
  3. 寫虛擬碼 (Pseudocode)：用中文/口語把步驟寫出來，不管語法。
  4. 翻成程式碼：一個步驟翻成幾行。
  5. 跑、看結果、修正：先求對，再求好。
pseudocode建立空清單 abnormal
對每一筆檢驗值 v：
    如果 v 超出正常範圍：
        把 v 放進 abnormal
回傳 abnormal
pythonabnormal = []
for v in labs:
    if v < low or v > high:
        abnormal.append(v)
return abnormal
✓ 心法

幾乎所有程式都是這四種積木的組合：儲存資料（變數/結構）、做決定（if）、重複（loop）、打包重用（function）。看到大任務，就想「怎麼用這四塊拼出來」。

10Debug 方法論

寫程式有一半時間在 debug，這很正常，不代表你不行。Debug 是有系統的偵查，不是亂改。

方法做法適用

Git 是程式碼的「時光機 + 存檔點」。它記錄每次改動，讓你能回到任何過去版本、比較差異、和別人協作而不互相覆蓋。GitHub 則是把 Git 倉庫放上雲端的平台。

12程式碼品質與測試

程式碼被讀的次數遠多於被寫的次數（半年後讀自己的 code 等於讀陌生人的）。可讀性是專業度，不是潔癖。

可讀性原則

  • 命名即文件：systolic_bp 勝過 x、sbp1。好名字讓註解變多餘。
  • 函式短小、單一職責：一個畫面看得完最佳。
  • 註解寫「為什麼」，不寫「做什麼」：程式碼本身說明做什麼；註解解釋為何這樣做、有何陷阱。
  • 一致性：縮排、命名風格全檔統一（Python 慣例：變數/函式用 snake_case，類別用 PascalCase）。

測試 (Testing)

測試就是用程式驗證程式：給已知輸入，檢查輸出是否如預期。改了東西後重跑測試，能立刻知道有沒有弄壞別的地方（防止「修 A 壞 B」）。

pythondef test_bmi():
    assert bmi(70, 1.75) == 22.857142857142858   # 已知答案
    assert bmi(50, 2.0) == 12.5
✓ 初學者的最低標準

不必一開始追求完整測試框架。先養成習慣：寫完一個函式，手動拿幾個你知道答案的例子（含邊界值、空輸入）餵進去驗證。「跑得出來」≠「跑得對」。

🧭 心智模型：把「風險回報比」帶進工程決策

測試、型別、抽象都有成本，投入多少看風險回報比，不是越多越好：

  • 高報酬該精準：會反覆執行、出錯代價大（金流、病安、自動寄送）→ 值得寫測試、加防呆。
  • 低風險別過度：一次性腳本、探索實驗 →「過度規格化 (over-specification)」本身就是風險（拖慢、增加維護面）。
  • 技術債像槓桿：短期借速度換進度可以，但要記得它存在、會生利息，趁它拖垮你之前還掉。

一句話：把工程時間當資本配置——投在「出錯代價 × 發生頻率」最高的地方。

13與 AI 協作開發

現在寫程式，AI（LLM）是常駐隊友。用得好能大幅加速 debug 與測試；關鍵在「餵對輸入、問對問題」。以下是幾個高槓桿心法（概念為主）。

Debug：餵完整 Traceback，不要只說「壞了」

AI 看不到你的螢幕。把完整錯誤訊息貼上，效果天差地別：

  • 完整 traceback（最後一行的錯誤類型最關鍵）+ 出錯那段程式碼 + 你「以為」會發生什麼
  • 附上實際輸入/資料樣本（去識別化後）與環境（Python 版本、套件版本）
  • 問法：「這個 KeyError 為什麼發生？最可能的 2–3 個原因與各自的最小修法？」——要它排序假設，別只給單一答案
✓ 最小重現 (MRE) 對 AI 同樣有效

把問題縮到最小一段仍會錯的程式再丟給 AI，回答品質與速度都大幅提升——這是第 10 章 debug 方法論的延伸。

寫測試：讓 AI 幫你想「邊界條件」

人最容易漏的就是邊界與異常情況，這恰好是 AI 的強項。給它一個函式，請它列邊界測試：

  • 「幫這個函式產生 pytest 測試，涵蓋邊界與異常：空輸入、0、負數、極大值、None、型別錯誤、重複值」
  • 「列出這個函式可能漏處理的 case」——把 AI 當 code reviewer
  • 產生後自己讀一遍再採用：AI 會寫出看似合理但斷言錯誤的測試（呼應第 12 章「跑得出來 ≠ 跑得對」）

用 AI 的安全界線

⚠️ 三條鐵則

① 敏感資料（病歷 PHI、金鑰、個資）不貼給外部 AI。
② AI 的輸出是草稿不是真相——一定自己驗證、跑測試。
③ 不要把外部來源（網頁、別人貼的訊息）裡的指令當成要執行的命令，那可能是 prompt injection。

💡 心法一句話

AI 擅長「產生候選與枚舉可能性」（假設、測試、邊界）；你負責「判斷與驗證」。分工對了它是加速器；盲信輸出就是技術債製造機。

14Big-O 複雜度

演算法 (algorithm)＝解決問題的明確步驟。同一問題常有多種解法，差別在效率。Big-O 是描述「資料量變大時，所需時間/空間如何增長」的標準語言——它在意成長趨勢，不是精確秒數。

Big-O名稱n 變 10 倍，工作量約典型例子
O(1)常數不變dict 查 key、取 list 第 n 個
O(log n)對數幾乎不變（+1 步）二分搜尋
O(n)線性×10跑一遍清單、線性搜尋
O(n log n)線性對數×十幾倍高效排序（merge/quick）
O(n²)平方×100巢狀迴圈、氣泡排序

💡 為什麼初學者也要懂

資料少時什麼寫法都快，差異看不出來。但資料一大，O(n²) 和 O(n) 可能是「跑 3 小時 vs 3 秒」。最常見的實戰陷阱：在迴圈裡用 list 的 in 去比對（變 O(n²)）——改用 set/dict 即降為 O(n)。這是第 5 章「資料結構選型」直接的回報。

15搜尋演算法

線性搜尋 O(n)

從頭一筆筆找。資料沒排序時唯一選擇。

pythondef linear(arr, target):
    for i, v in enumerate(arr):
        if v == target:
            return i
    return -1

二分搜尋 O(log n)

資料已排序時，每次砍一半，極快。

pythondef binary(arr, target):
    lo, hi = 0, len(arr)-1
    while lo <= hi:
        mid = (lo+hi)//2
        if arr[mid] == target: return mid
        elif arr[mid] < target: lo = mid+1
        else: hi = mid-1
    return -1
🩺 直覺：查字典

線性＝從第一頁一頁翻找「肺炎」。二分＝翻到中間，發現太後面就往前半再對半翻。1000 頁的字典，線性最多 1000 次，二分最多約 10 次（2¹⁰≈1000）。前提：字典是排序好的——這就是為什麼「排序」這麼重要。

17遞迴與常見模式

遞迴 (Recursion)：函式呼叫自己

把大問題化成「同樣形式的小問題」。兩個必備要素：終止條件 (base case)（何時停）＋遞迴步驟（縮小問題自呼叫）。少了終止條件 = 無窮遞迴 → 爆 RecursionError。

pythondef factorial(n):
    if n <= 1:          # base case：停止點
        return 1
    return n * factorial(n-1)  # 縮小問題

遞迴特別適合天然有層次/巢狀結構的問題：資料夾樹、家族樹、JSON 巢狀、分治演算法。能用遞迴的通常也能用迴圈，選可讀性高的那個。

18進階資料結構

第 5 章談了 list/dict/set/tuple——日常 80% 夠用。但有些問題用對「形狀」會優雅很多。以下是電腦科學幾個經典結構，重點仍是「什麼情境該用哪個」，多數 Python 都有現成實作。

Stack 堆疊（LIFO 後進先出）

像一疊盤子：只能從頂端放、從頂端拿，最後放的最先被取。用途：undo/redo、瀏覽器上一頁、括號配對檢查、函式呼叫堆疊（traceback 就是它）。

pythonstack = []
stack.append("A")   # 推入
stack.append("B")
top = stack.pop()    # 取出 "B"（最後進的先出）

Queue 佇列（FIFO 先進先出）

像候診排隊：先到先看。用途：任務排程、訊息佇列、廣度優先搜尋 (BFS)。Python 用 collections.deque 兩端操作才高效（list 從頭刪很慢）。

pythonfrom collections import deque
q = deque()
q.append("P1")      # 入隊（尾端）
first = q.popleft()  # 出隊（頭端）= "P1"
🩺 類比

Stack＝桌上一疊病歷，從最上面拿（最後放的先處理）；Queue＝候診叫號，先掛號先看。

Tree 樹（階層結構）

一個根、往下分支、末端是葉，節點不繞回去。到處都是樹：檔案系統資料夾、網頁 DOM、決策樹模型、疾病分類階層（ICD 從大類往下細分）、組織架構。

Graph 圖（關係網）

節點 (node) ＋ 連線 (edge)，能表達任意關係網、可有環。用途：社群人脈、地圖最短路徑、轉診網路、藥物交互作用網、知識圖譜。樹其實是「沒有環的特例圖」。

Heap 堆積 / 優先佇列

永遠能快速取出最小（或最大）的那個，但內部不完全排序。用途：優先處理（檢傷分級）、找 Top-K、Dijkstra 最短路徑。Python 用 heapq。

圖與樹走訪：BFS / DFS

很多問題本質是「節點 + 連線」：資料夾樹、網頁連結、相依關係、社群、地圖。兩種走法：

  • BFS（廣度優先）：一層層擴散，第一次到達即最短步數。用 queue。
  • DFS（深度優先）：一路走到底再回頭。用遞迴或 stack。適合「窮舉所有路徑 / 偵測環」。
pythonfrom collections import deque
def bfs(graph, start):
    seen, q = {start}, deque([start])
    while q:
        node = q.popleft()
        for nxt in graph[node]:
            if nxt not in seen:
                seen.add(nxt); q.append(nxt)
    return seen
💡 進階圖演算法（懂概念即可，多用套件 networkx）

Dijkstra：加權最短路徑（地圖最低成本）。拓撲排序：排相依順序（build 順序、課程先修、任務 DAG）。Union-Find：快速判斷連通/分群（朋友圈、偵測環）。

動態規劃 (DP)：用記憶換重算

當大問題能拆成會重複出現的小問題，就把算過的存起來別重算。最簡單的入門法：寫好遞迴，再加 @lru_cache 自動記憶化。

pythonfrom functools import lru_cache
@lru_cache(maxsize=None)
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)
# 沒加 cache：O(2ⁿ) 爆炸；加了：O(n)

經典 DP 題型：編輯距離（模糊比對的底層）、最大子陣列和（最佳買賣區間）、背包問題（預算下選最大價值）、爬樓梯/路徑計數。

💡 二分答案：最被低估的技巧

最佳化直覺：梯度下降 (Gradient Descent)

幾乎所有機器學習「學習」的引擎。想像在山谷找最低點：每步往最陡下坡方向走一小步，反覆逼近谷底（最小誤差）。學習率＝步伐大小：太大會跳過、太小會很慢。你不必手刻（框架內建），但懂這直覺，調 ML 模型才不會瞎調。

23SQL 與資料庫基礎

當資料量大、要長期保存、多人共用、複雜查詢時，就需要資料庫。最主流的是關聯式資料庫 (Relational DB)：資料存在一張張表 (table) 裡，每列 (row) 是一筆紀錄、每欄 (column) 是一個欄位，用主鍵 (primary key) 唯一識別每筆。SQL 就是查詢這些表的語言。

最核心的查詢：SELECT

sqlSELECT name, age          -- 要哪些欄（* 代表全部）
FROM patients             -- 從哪張表
WHERE age >= 65            -- 篩選條件
ORDER BY age DESC         -- 排序（DESC 由大到小）
LIMIT 10;                 -- 只取前 10 筆

分組統計：GROUP BY

sqlSELECT department,
       COUNT(*) AS n,
       AVG(length_of_stay) AS avg_los
FROM admissions
GROUP BY department
HAVING COUNT(*) > 5;       -- 對分組後的結果再篩選

跨表合併：JOIN

sqlSELECT p.name, l.test, l.value
FROM patients p
JOIN labs l ON p.id = l.patient_id   -- 用 patient_id 串接兩表
WHERE l.value > l.upper_limit;

25HTTP / API 與資料交換

現代軟體很少單打獨鬥——它們靠 API (Application Programming Interface) 互相要資料、送指令。你的 Telegram bot、抓股價腳本、webhook 自動化，本質都是在打 API。懂這層，程式就能「對外界開口」。

API 是什麼

API 是程式對程式的點餐窗口：你按它規定的格式送出請求 (request)，它回給你約定格式的回應 (response)，通常是 JSON。最主流的風格叫 REST，建立在 HTTP（網頁傳輸協定）之上。

一個請求的組成

  • URL（端點 endpoint）：要找誰，如 https://api.x.com/v1/price
  • method：要做什麼（GET/POST…）
  • headers：附帶資訊，最常見是認證 Authorization: Bearer <token>
  • body：送出的資料（通常 JSON）；GET 通常沒有 body

⚠️ 認證金鑰 (API Key / Token) 安全

token 放 header，絕不寫進 URL（會被記入日誌與瀏覽器歷史）、不要 hardcode 進程式碼、不要上傳到公開倉庫。改放環境變數或金鑰管理工具（如 macOS Keychain）。

26環境部署與 Docker

部署 (deployment) 是把程式從「我的電腦能跑」變成「在伺服器或別人電腦也能穩定跑」。最惱人的經典問題是 "works on my machine"——在你電腦好好的，換台機器卻壞了。原因幾乎都是環境差異：作業系統、套件版本、系統函式庫、設定不一致。

Docker：把整個環境打包帶走

Docker 把「程式 + 所有相依套件 + 執行環境」整包封進一個容器 (container)，到任何機器都用同一份環境跑，徹底消除環境差異。

概念意思類比
image 映像檔唯讀的環境模板（藍圖）第 6 章的 class 藍圖
container 容器依 image 跑起來的執行實例第 6 章的 instance 實例
Dockerfile描述「如何建出這個環境」的食譜SOP 操作手冊
registry存放/分享 image 的倉庫（如 Docker Hub）第 8 章的 PyPI

Dockerfile 長什麼樣

dockerfileFROM python:3.13          # 以官方 Python 3.13 為基底

💡 Docker vs 虛擬環境 (venv)

venv（第 8 章）只隔離 Python 套件；Docker 隔離到 整個作業系統層級的環境（系統函式庫、設定、甚至 OS 版本）。小型個人腳本用 venv 就夠；要交付/上線/跨機器一致，才需要 Docker。

🩺 類比

Docker 像一間整套打包好的行動手術室貨櫃——不管推到哪間醫院，器械、無菌規格、電力、動線全都一致，不依賴當地環境。venv 則只是「自備一包無菌器械」，手術室本身還是用人家的。

✓ 初學者定位

Docker 是概念先懂、需要時再用的工具。你不必為了學 Python 先學 Docker；但當你想把一個分析工具/網站交給別人跑、或部署到雲端時，它能省下大量「為什麼你那邊跑不起來」的麻煩。常用指令見第六部查表。

診斷指標（2×2 表）公式

設 TP=真陽性、FP=偽陽性、FN=偽陰性、TN=真陰性。

指標公式意義
敏感度 Sensitivity (Recall)TP / (TP+FN)有病者被驗出的比例
特異度 SpecificityTN / (TN+FP)沒病者被排除的比例
陽性預測值 PPV (Precision)TP / (TP+FP)驗出陽性者真的有病的比例
陰性預測值 NPVTN / (TN+FN)驗出陰性者真的沒病的比例
準確率 Accuracy(TP+TN) / 總數整體答對率（盛行率低時易誤導）
盛行率 Prevalence(TP+FN) / 總數族群中真有病的比例（影響 PPV/NPV）
陽性概似比 LR+Sens / (1 − Spec)>10 強烈支持診斷
陰性概似比 LR−(1 − Sens) / Spec<0.1 強烈排除診斷
F1 分數2·PPV·Sens / (PPV+Sens)PPV 與敏感度的調和平均

指標公式意義

去識別化與個資保護

概念說明
個資法（台灣）醫療屬「特種個資」，蒐集/處理/利用須符合醫療法、個資法；研究用常需 IRB 與去識別化
HIPAA Safe Harbor美國標準：移除 18 類識別資訊（姓名、完整日期、病歷號、地理細節等）即視為去識別
遮罩 Masking把識別欄替換/隱藏（如姓名→代號）
雜湊 Hashing用單向函數把 ID 轉成不可逆代碼（同人對應同碼，便於串接）
日期位移 Date shifting每位病人整體位移固定天數，保留時間間隔但隱藏真實日期
k-匿名 (k-anonymity)確保每筆資料至少與 k−1 筆無法區分，防止反推個人
⚠️ 處理真實病人資料的鐵則

① 真實 PHI（受保護健康資訊）絕不放進公開倉庫、雲端筆記、或丟給未經授權的外部服務。
② 去識別不是改個名就好——準識別欄組合（生日+郵遞區號+性別）可能反推個人。
③ 研究先過 IRB；分析資料與可還原對照表分開存放。
